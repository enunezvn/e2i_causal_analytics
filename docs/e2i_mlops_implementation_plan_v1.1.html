‚ö° Key Architecture Principle: Decouple Models from Agents
‚ùå Don't: Load models in agent process
  • ‚Ä¢ Agent RAM: 6GB+ per instance
  • ‚Ä¢ Model OOM crashes entire agent
  • ‚Ä¢ Cold start: 30-60 seconds
  • ‚Ä¢ Can't scale independently
‚úÖ Do: Models as separate services
  • ‚Ä¢ Agent RAM: ~500MB (lightweight)
  • ‚Ä¢ Fault isolation (model ‚â† agent)
  • ‚Ä¢ Agents call endpoints via tools
  • ‚Ä¢ Scale models & agents separately

Schema Decision: Extend Existing
Add 8 new MLOps tables to existing V3 schema (18 ‚Üí 26 tables)

‚ùå Option A: Separate MLOps Schema

Create a new mlops schema alongside public schema

  • ‚úó Cross-schema joins are complex
  • ‚úó Duplicates split enforcement logic
  • ‚úó Harder to maintain referential integrity
  • ‚úó More migration complexity

Option B: Extend Existing Schema

Add MLOps tables to existing public schema with ml_ prefix

  • ‚úì Unified split enforcement
  • ‚úì Simple foreign keys to existing tables
  • ‚úì One migration path (007_mlops_tables.sql)
  • ‚úì Existing repository pattern works
