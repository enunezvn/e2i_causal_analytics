# docker-compose.dev.yml - Bind Mounts for Development
services:
  api:
    volumes:
      # Bind mounts for hot-reloading
      - ./src:/app/src                    # Python source
      - ./config:/app/config              # YAML configs
      - ./tests:/app/tests                # Test files
    command: uvicorn src.api.main:app --reload --host 0.0.0.0
    
  frontend:
    volumes:
      - ./frontend/src:/app/src           # React source
      - ./frontend/public:/app/public     # Static assets
    command: npm run dev -- --host 0.0.0.0

ğŸ’¡ Best Practices & Gotchas

âœ… DO: Separate Dev & Prod Compose Files
Use docker-compose.yml for production (no bind mounts) and docker-compose.dev.yml with bind mounts. Run dev with:
docker compose -f docker-compose.yml -f docker-compose.dev.yml up
âœ… DO: Use Read-Only Where Appropriate
BentoML only reads models, never writes. Mount as :ro to prevent accidental corruption and improve security.
âŒ DON'T: Bind Mount node_modules or __pycache__
These should stay in the container. Use a named volume or .dockerignore to exclude them. Otherwise, OS-specific binaries will conflict.
âŒ DON'T: Share Volumes Across Incompatible Writes
If two containers write to the same file (not directory), you'll get race conditions. Use separate subdirectories or message queues instead.
ğŸ’¡ TIP: Use tmpfs for Ephemeral Computation
For temporary causal computation scratch space, use tmpfs mounts. Faster than disk and auto-cleans on container restart.
ğŸ’¡ TIP: Volume Initialization Pattern
For shared volumes that need initial data (like Feast feature definitions), use an init container or entrypoint script that checks if empty before populating.
